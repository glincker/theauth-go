# v0.6 hardening retrospective

## What shipped

Five workstreams landed in `v0.6/hardening`:

1. **Fuzz tests (WS1):** 11 fuzz targets across `crypto/`, the root
   package, and the OAuth handlers. Local 15 to 30 second runs and CI
   10 second per target runs are wired.
2. **Race tests (WS2):** five concurrency tests for the OAuth state map,
   the rate limiter (same and different IPs), and session creation plus
   lookup under contention.
3. **Examples (WS3):** four new runnable apps (gin, echo, stdlib,
   oauth-multi-provider). Each ships README, single file `main.go`,
   `go.mod`, `docker-compose.yml`, `.env.example`, and `Makefile`. The
   existing `chi-app` got the same scaffolding for parity.
4. **Benchmarks (WS4):** five benchmarks under `internal/bench/`
   measuring password sign in, session lookup, OAuth callback storage
   cost, magic link request round trip, and PKCE challenge derivation.
   Baselines recorded in `internal/bench/BASELINES.md`.
5. **Docs and stability (WS5):** `doc.go` in every package, runnable
   `Example` functions for the five most-used entry points,
   `STABILITY.md` enumerating the v1.0 contract, `CHANGELOG.md`.

## Bugs found by fuzzing

One harness bug: `FuzzCallbackQueryParams` initially built the test
request through `httptest.NewRequest`, which panics on a URL containing
a raw space. Switched to direct `http.Request` construction. The
captured failing input is committed under
`testdata/fuzz/FuzzCallbackQueryParams/930f49fab2367014` as a
regression case.

No bugs found in production code.

## Baselines (Apple M4 Max, Go 1.25)

See `internal/bench/BASELINES.md` for the raw output and methodology.
The Argon2id heavy `BenchmarkLoginPassword` clocked in well under the
spec's expected M2 baseline because the test machine is roughly 3 to 4
times faster than an M2 for CPU bound work. Argon2id parameters
themselves are unchanged.

## Deviations from spec

- Email validator was added as a fresh helper (`validateEmail`) rather
  than discovered as a pre-existing function, since the codebase used
  `mail.ParseAddress` inline. The helper is exposed via
  `ValidateEmailForTest` in `export_test.go`, matching the spec's
  pattern.
- The benchmarks live under `internal/bench/` per spec but reach the
  hot paths through the public HTTP handlers rather than via direct
  service-method calls. This keeps the bench package portable (no
  dependence on test-only exports across package boundaries) without
  sacrificing fidelity: the timed region still includes the JSON parse,
  argon2 verify, and session create steps.
- No deprecation markers were added. The spec listed
  `Config.LegacyCookieName` and `Config.AllowInsecureCookies` as
  potential deprecation candidates; neither exists in the tree, so the
  audit recorded "no deprecations" in STABILITY.md and moved on.
